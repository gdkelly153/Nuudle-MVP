# Puzzle Generation Framework Update Plan

This document outlines the plan to update the puzzle generation framework based on the analysis of lateral thinking puzzles.

## 1. New Puzzle Generation Logic

The new puzzle generation logic will be based on a scenario-driven algorithm that ensures alignment with our quality characteristics. The process will be as follows:

1.  **Scenario Selection:** The generator will select a common, easily understood scenario (e.g., "a man is found dead in a room with a puddle of water").
2.  **Core Riddle Generation:** A concise riddle will be generated by removing key contextual details from the scenario. The goal is to create ambiguity while keeping the language simple.
3.  **Solution Definition:** The solution will be broken down into distinct **Solution Components**. For the "Man with the Straw" puzzle, these would be:
    *   Component 1: The man was in a hot air balloon.
    *   Component 2: The balloon was overweight and failing.
    *   Component 3: He and a companion drew straws to decide who would jump.
    *   Component 4: He drew the short straw and sacrificed himself.
4.  **Icon/Keyword Generation:** For each Solution Component, the system will generate a corresponding icon or a single, evocative keyword (e.g., "Balloon", "Weight", "Choice", "Sacrifice"). This will serve as the visual representation of the clue on the card.
5.  **Semantic Analysis:** The riddle and each solution component will be analyzed for clarity and coherence. The framework will now support multi-part solutions, as long as each component is logical and concise.

## 2. Clue Component Strategy

The clue generation and selection process will be redefined to be more dynamic and strategic. We will move away from a fixed number of clues and toward a system where clues are chosen for their strategic value.

1.  **Clue-to-Component Mapping:** Each clue will be directly mapped to one of the solution components and its corresponding icon/keyword.
2.  **Dynamic Unlocking:** When a user's guess is close to a solution component, the system will unlock the corresponding icon/keyword. This provides targeted, visual feedback that reinforces their line of thinking without cluttering the UI.

## 3. UI/UX Persistence

To ensure clue cards remain visible after an incorrect answer, the following state management changes are required:

1.  **Icon/Keyword Display:** The clue cards will be updated to display the icon or keyword for each unlocked solution component. This maintains the clean, minimalist design while providing meaningful progress indicators.
2.  **`unlockedComponents` State Management:** A state variable, `unlockedComponents`, will store the icons/keywords that have been revealed. This state will persist across incorrect guesses, ensuring the user's progress is always visible.
3.  **Tooltip for Detail:** To provide more context without increasing card size, a tooltip will appear when the user hovers over an unlocked clue card, revealing the full text of the clue (e.g., "Hovering over the 'Balloon' icon shows: 'The man was in a hot air balloon.'").

## 4. Implementation Roadmap

The implementation will be broken down into the following phases:

**Phase 1: Backend Development**
*   **Task 1.1:** Implement the new scenario-driven puzzle generation logic.
*   **Task 1.2:** Develop the tiered clue generation and strategic value scoring system.
*   **Task 1.3:** Update the API to expose the new puzzle and clue structure to the frontend.

**Phase 2: Frontend Development**
*   **Task 2.1:** Implement the `revealedClues` state management for UI persistence.
*   **Task 2.2:** Update the UI to dynamically display clues based on the tiered system.
*   **Task 2.3:** Integrate the updated API endpoints for fetching puzzles and clues.

**Phase 3: Testing and Deployment**
*   **Task 3.1:** Conduct end-to-end testing of the new puzzle generation and display logic.
*   **Task 3.2:** Gather user feedback on the quality of the new puzzles and the clue system.
*   **Task 3.3:** Deploy the updated framework to production.
